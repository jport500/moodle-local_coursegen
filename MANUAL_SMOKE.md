# local_coursegen — manual smoke test

An end-to-end check that a real install can take source material to a completed,
hidden course that a learner can finish. Run it on a site with the full stack
(format_pathway, local_quizgenpro, mod_knowledgecheck + filter_knowledgecheck)
and a configured core_ai text (and, for images, image) provider.

The automated suite (`vendor/bin/phpunit`) covers the server paths; this script
covers the things only a human at a browser can confirm — the AI Providers
transport, the inline knowledge-check render, and the learner completion
round-trip.

## 0. Preconditions

- [ ] **core_ai text provider configured and enabled.** Site admin → AI →
  Provider settings. Without an enabled text provider, blueprinting fails with
  "no provider enabled". (For section images, also enable an image provider.)
- [ ] **filter_knowledgecheck is enabled** (Site admin → Plugins → Filters →
  Manage filters → "Knowledge check" = On). If it is off, assessed sections fall
  back to a **non-stealth** check on the course page with no inline render (by
  design — never a silently invisible check), and this smoke can't verify inline
  rendering. See the footguns below.
- [ ] **Spend cap is sane.** Site admin → Plugins → Local plugins → Course
  generator. Confirm `cap_period_spend` is not the old `50` (see footguns). A
  fresh install or a plugin upgrade from a pre-P6 build self-corrects an
  untouched `50`; verify anyway.

## 1. Create a job and ingest source

Either through the UI (`/local/coursegen/index.php` → upload a file or enter a
topic) or via CLI:

```
php local/coursegen/cli/extract.php --file=/path/to/source.pdf --categoryid=<catid>
# or
php local/coursegen/cli/extract.php --topic="Introduction to widgets" --categoryid=<catid>
```

- [ ] Note the printed **job id**.
- [ ] Job status reaches `extracted`.

## 2. Blueprint (real provider call)

```
php local/coursegen/cli/blueprint.php --jobid=<id>
```

- [ ] "Resolved text provider" names your enabled provider (not "(none
  enabled!)").
- [ ] Result is `success`; status is `awaiting_review` (outline-first) or
  `approved` (automatic mode).
- [ ] The printed estimate (generation units) and section list look plausible.

## 3. Review gate (outline-first)

Optionally regenerate a section, then approve:

```
php local/coursegen/cli/gate.php --jobid=<id> --regen=0      # optional
php local/coursegen/cli/gate.php --jobid=<id> --approve
```

- [ ] After approve, status is `approved` (or `materializing`/`complete` if the
  adhoc task has already run).
- [ ] A post-approval edit/regen reopens the job to `awaiting_review`.

## 4. Materialize (real provider calls)

Either let the queued adhoc task run (`php admin/cli/adhoc_task.php
--execute=...`) or run inline:

```
php local/coursegen/cli/materialize.php --jobid=<id>
```

- [ ] Result `success`; status `complete`; a course id is printed with
  `visible=0` (the course is created **hidden**).
- [ ] "Actual spend (generation units)" is recorded and is within the cap.

## 5. Inspect the generated course (browser, as admin)

Open the new course.

- [ ] One section per blueprint section, each with inline reading text (a "Text
  and media" area) and, where opted in, an inline image with alt text.
- [ ] Assessed sections render an **inline knowledge check in the reading flow**
  (not a separate activity link) — this is filter_knowledgecheck expanding the
  `{knowledgecheck id=…}` token. If you instead see a visible "Knowledge check"
  activity on the course page, the filter is off (see preconditions).
- [ ] The knowledge check has real questions (generated by local_quizgenpro).

## 6. Learner completion round-trip (the point of the whole thing)

1. Enrol a test learner (or use "Log in as") and make the course visible, or test
   as an enrolled learner while the course is hidden via a role that can see
   hidden courses.
2. As the learner, open a section and **submit the inline knowledge check**.

- [ ] The activity flips to **complete** (completion tick).
- [ ] **Pathway progress** for the course advances (format_pathway progress bar).
- [ ] With an activity-completion criterion configured, **course completion**
  registers after the completion cron (`php admin/cli/cron.php` or the
  `completion_regular_task`).

This is the behaviour the automated `completion_walkthrough_test` exercises at
the server layer; step 6 confirms the browser → webservice → completion path that
the test can't drive.

## 6b. Rebuild protection for a live course (no wrap needed)

A re-materialize is delete-and-rebuild, so the guard refuses whenever it would
destroy live learner state — including a learner who reached the course by direct
enrolment, with no program/certification involved.

1. Unhide a generated course and **directly enrol a learner** (Participants →
   Enrol users), or have one complete an activity.
2. Edit a section and **re-approve** the job to trigger a re-materialize.

- [ ] It is **refused** — the job **stays COMPLETE**, the course and the learner's
  enrolment/completion are untouched, and a `coursegen_log` row (outcome `failure`)
  names the directly-enrolled learner / completion count.
- [ ] **Retry after clearing:** unenrol the learner (and clear any completion),
  then edit + re-approve again — the rebuild now proceeds.

## 7. Cert-chain wrap (optional — needs tool_muprog / tool_mucertify)

In the plugin settings, turn on **Wrap in program** and **Wrap in certification**,
then materialize a job (steps 1–4).

- [ ] A `tool_muprog` program with idnumber `coursegen-job-<jobid>` exists and
  contains the generated course (Site admin → Programs).
- [ ] A `tool_mucertify` certification with the same idnumber exists, linked to
  that program (Site admin → Certifications).
- [ ] **Retry idempotency (empty wrap):** re-run materialize for the same job
  (e.g. reopen and re-approve, or re-run the task) *before* allocating anyone.
  Exactly **one** program and **one** certification with that idnumber remain — no
  duplicates, no stranded artifacts.
- [ ] **Populated-wrap protection:** now allocate a learner to the program (configure
  an allocation source and allocate a user), then edit + re-approve to re-run
  materialize. It is **refused** — the job **stays COMPLETE** (the live course is
  untouched) with a `coursegen_log` row (outcome `failure`) naming the program and
  the allocation count, and the existing course, program, certification and the
  learner's allocation are **all still there**.
- [ ] **Retry after clearing:** clear the allocation (or detach the program), then
  edit + re-approve again — the wrap is now empty, so the rebuild proceeds and the
  course/program/certification are rebuilt fresh.
- [ ] No learners are auto-enrolled — the wrap builds containers only.

## 8. Spend cap (optional, destructive to the period total)

- [ ] Lower `cap_period_spend` below the current period total, then attempt a new
  blueprint/regen/materialize: it is **refused before any AI call** and audited
  in `coursegen_log` (`outcome=failure`, detail mentions the cap).
- [ ] Restore `cap_period_spend`.

---

## Governance footguns

- **Don't disable filter_knowledgecheck on a tenant that already has generated
  courses.** Existing inline `{knowledgecheck id=…}` tokens stop rendering and the
  embedded checks vanish from the reading flow. New builds fall back gracefully;
  existing ones do not re-render until the filter is back on.
- **The old `cap_period_spend=50` default is a silent hard-stop.** Installs
  created on a pre-P6 build defaulted to 50 generation units — far below a single
  course's token-scale estimate — so generation refuses with nothing obviously
  wrong. The P6 upgrade bumps an *untouched* 50 to 1000000 automatically; if an
  admin deliberately set 50 it is left alone. If generation refuses unexpectedly,
  check this value first.
- **`cap_period_spend` / `cap_image_count` of 0 means unlimited**, not "blocked".
  Set a real number to cap; leave 0 only when you intend no ceiling.
- **The spend cap is a rolling window** (`period_days`, default 30). Spend ages
  out of the total, so a tenant blocked today may proceed later as old entries
  fall outside the window. This is intended; set `period_days` to match your
  billing period.
- **core_ai routes by provider order, not per-call selection.** coursegen cannot
  pin a specific provider for a call; whichever enabled provider is first for the
  action is used. Order your providers deliberately.
- **Generated courses are always created hidden.** Nothing is exposed to learners
  until an admin reviews and unhides it.
- **The muprog/mucertify "wrap" toggles build containers, not enrolments.** With
  "Wrap in program" on, a materialized course is added to a new tool_muprog
  program; with "Wrap in certification" also on, a certification is linked to that
  program. Neither allocates learners — configure the program's allocation sources
  afterwards. "Wrap in certification" requires "Wrap in program" (the UI hides it
  otherwise). If a toggle is on but tool_muprog/tool_mucertify is absent, the wrap
  is skipped with a logged warning; the course still builds.
